## 2026-07-21 — params plumbed to the vendor call (F1 working in F3's lane, waived by Poulav)

**Lane note:** apps/firm is F3's lane. Poulav explicitly waived the boundary for
this change because G2 is blocked on it and submission is tomorrow. Ishita: this
is yours to review or revert; nothing else in apps/firm was touched.

The bug: `_procure_subtask` sent `args={"goal": ..., "subtask": ...}` to every
vendor. Real marketplace vendors have real schemas — OKLink #2023 requires
`chainIndex`, `address`, `height` — and the procurer pays BEFORE the vendor
validates the body. So a live call with a generic body is not a soft failure,
it is a paid-for 400.

The fix, kept inside the frozen contract:
- `FirmTask.params`, new `firm_jobs.params` column (migration 004), plumbed
  through `_save_task`, `get_task`, `claim_next_task`, and `claim_task`.
- `_vendor_args(task, subtask)`: sends job params verbatim and alone when
  present; otherwise preserves the previous generic shape, which is what the
  packages/mocks fixtures expect.
- `express_run` already parsed `params` per INTERFACES §1A and then **discarded
  them**. It now persists them. No contract change was needed.

Deliberately NOT done:
- No params on `execute`. INTERFACES §1B defines it as `{quote_id}` only, and
  widening a frozen contract needs both humans. Firm Projects therefore still
  cannot drive a schema-bearing vendor; that is a real limitation and a
  proposal, not something I patched around.
- No `FirmTask.constraints`. I started to add one before noticing constraints
  already ride on `quote.constraints` and are read there by `sourcing_node`.
  Two sources of truth for a buyer's filters would be a bug generator. Reverted.

A note on the test double: `tests/test_storage.py`'s FakeCursor positionally
destructures INSERT params, so adding a column silently shifted every field
after it — `progress` received `{}` and `deliverable` received the progress
list. Seven tests caught it immediately, which is the double earning its keep,
but the positional destructuring is fragile and worth revisiting.

## 2026-07-21 — never pay a vendor whose declared params we cannot supply

(apps/firm, F3's lane. Same waiver from Poulav as the params change. Ishita: this
is yours to review.)

`VendorService` had no `documented_example_args` field, so Pydantic silently
dropped what tools/vendor-index had carefully recorded. The worker therefore
could not see a vendor's declared request schema and would pay a vendor it was
guaranteed to fail with — the payment happens before the vendor validates the
body, so that is money spent on a certain 400.

Now: `missing_documented_params(service, args)` compares the vendor's own
published example against what the job supplies, and `_procure_subtask` runs it
BEFORE `pay_and_call`. A vendor we cannot satisfy is skipped for free.

Two design points worth keeping:

1. **Skipped, not fired, and no performance penalty.** A vendor whose documented
   params we did not supply has not failed — we have. Recording that as a firing
   would repeat exactly the bug that fired OKLink this morning for delivering
   correctly, and would put another false accusation in a provenance receipt. It
   is recorded as a rejection with an honest reason: "not hired: job supplies no
   address, chainIndex, height, which this vendor documents as required".

2. **Deliberately strict.** Every documented key is treated as required. We
   cannot distinguish an optional key from a mandatory one in a published
   example, and the two errors are not symmetric: too strict skips a vendor and
   costs nothing, too lax spends real money on a call that cannot succeed.

A vendor that documents nothing returns [] and is unaffected — unknown is not a
failure, and most of the marketplace documents nothing.

Verified behaviourally, not just by unit test: with no params, 0 payments made,
0 firings, no performance record written, honest rejection recorded. With params
supplied, the same vendor is paid and hired.

Verified: 38 pytest, 64 procurer, 28 gateway, 10 evals, 6 vendor-index.

## 2026-07-21 — a simulated books call no longer reduces our stated margin

`build_provenance` subtracted a flat 50,000-unit books cost from margin whether
or not the Treasury call happened. Treasury is not listed, so it never happens:
`books.tx` has always read `SIMULATED:treasury-books`.

The effect on the G2 receipt: we reported margin 49,985 when we had actually
retained 99,985. **Understating our own margin against a cost we never incurred**
— dishonest in the generous direction, which is still dishonest, and on the
receipt this entry asks judges to trust.

Now the books line is disclosed either way but only counts as an incurred cost
when `ENABLE_TREASURY_BOOKS` is on. While it is off, cost is 0 and the statement
says plainly that no call was made and no cost was incurred. A test pins
`books == 0` alongside the existing reconciliation identity.
