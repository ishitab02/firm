## 2026-07-21 — real and simulated spend are now separate ledgers

Found while dry-running G2: the run failed with `CAP_EXCEEDED` even though only
15 base units of real money had ever been spent. Cause: **simulated and real
payments shared one spend ledger.** `spent_today` read 1,200,015 — 15 units of
actual money plus 1.2M of eval and demo traffic.

Two ways that bites, and the first one actually happened:
1. A day of tests exhausts the real daily cap and blocks a real payment.
2. `/caps` — the operator's view of money — reports simulated spend as real.

Fix: `procurer_calls.mode` ('real' | 'simulated'). Cap aggregation filters by
mode, so each ledger has its own budget and tests still prove cap enforcement
against their own. `/caps` reports `spent_today` (real only) and
`simulated_today` separately. Existing rows were backfilled from the recorded
receipt rather than guessed — a `SIMULATED:` tx was never real money. Result on
this machine: 20 simulated rows, 1 real row, correctly classified.

An idempotency key that already holds a record in one mode now REFUSES a request
in the other, rather than replaying across. Returning a settled simulated receipt
to a real caller would look exactly like a completed payment that never happened.

Four tests pin it. Verified: 64 procurer, 27 pytest, 28 gateway, 10 evals.

**Latent footgun found, NOT fixed, deliberately.** `reserveCall` takes both a
`ceiling` object (persisted as the reservation) and a `ceilingUnits` number (used
for cap math). Production always derives both from `max_amount` so they agree,
but nothing enforces it — and in the test fixtures they diverge, which is what
made my first ledger test silently record 0. Deriving units from the ceiling
inside `reserveCall` would remove the whole class of bug, but it changes the
signature and several existing tests deliberately pass mismatched values, so
rewriting them hours before submission is the riskier move. Recorded for after.

## 2026-07-21 — pre-payment vendor vetting, and a marketplace health scan at real scale

### Why

The README claims The Firm "background-checks vendors before paying". It did
not. `vetting_node` in the worker logs a state transition and returns; the only
thing standing between a bad vendor and our wallet was the cap check inside
`/pay-and-call`, which fires after we have already chosen that vendor and
started the call. That is a real claim-versus-code gap, and it is the kind a
reviewer finds by reading one function.

The other half: the reliability research (D4) produced the one genuinely novel
dataset we have, and it was a throwaway script. Research that cannot be re-run
is an anecdote.

### What landed

`packages/procurer/src/vet.ts` — an unpaid probe that reads a vendor's 402 and
reports what hiring them would actually cost. Seven verdicts: `X402_OK`,
`PRICE_MISMATCH`, `OVER_BUDGET`, `NO_CHARGE`, `UNSUPPORTED_CHALLENGE`,
`HTTP_ERROR`, `UNREACHABLE`. Exposed at `POST /vet` (single or batch) and as
`pnpm -F @firm/procurer vet`.

It shares `parseChallenge`/`selectOffer` with the money path, so the vet reads a
challenge exactly the way the payer would. I deliberately did **not** refactor
`vendor.ts` to share the fetch plumbing: the interpretation is what has to
match, and that already lives in `x402.ts`. Touching the money path for tidiness
three days before submission is not a trade I would take.

Two properties I wanted pinned rather than assumed:

- **Vetting cannot spend.** `vet.ts` never imports the signer and never reads a
  key. There is a structural test asserting that against the source text, because
  this is the one place that talks to vendor endpoints without a cap check in
  front of it, and that is only safe because it cannot sign.
- **Vetting is not an authorisation.** `hireable: true` narrows the candidate
  list; it never widens what may be paid. A vendor can reprice between the probe
  and the call, so `/pay-and-call` still re-reads the challenge and re-checks
  every cap against the amount it is about to sign. Both facts are in the module
  comment, so nobody later "optimises" the second probe away.

### Validation: the product code reproduces the hand research

Ran it against the same 10 agents D4 probed by hand. **9 of 10 verdicts match
exactly.** The tenth is a finding, not a discrepancy — see below.

### The retry, and why it exists

SignalForge #6560 was recorded UNREACHABLE by the hand research. The product
code reached it. Probing directly three times: fail, 402, 402.

It is not dead, it is cold — free hosting spins containers down. A single probe
cannot distinguish "dead" from "cold-starting", and condemning a live vendor on
one timeout is the same class of error as the validator bug that libelled OKLink.
So `vetVendor` retries **network-level failures only**, twice by default, and
reports `attempts` so a flaky vendor stays visibly flaky. A 402 or a 404 is a
definitive answer and gets exactly one shot — retrying those would triple our
traffic against every dead endpoint on the marketplace for no information.
Both directions are tested.

SignalForge also charges **50x its listing** (10,000 listed, 500,000 live). So
the mispricing count went from one to two before the scan even widened.

### Marketplace health scan — 95 agents

`data/marketplace-health-2026-07-21.json`, regenerable with:

    pnpm -F @firm/procurer vet -- --index data/marketplace-scan.json --out <path>

Population: of 218 scanned agents, **95 publish an A2MCP service with an HTTP
endpoint at all**. The other 123 are A2A-only or serviceless — nothing to probe.
One unpaid POST to the first endpoint-bearing service per agent.

- **41 of 95 (43%) dead or misrouted** at their listed endpoint
- 54 hireable
- **5 charge above their listing**: Clawby 600x, SignalForge 50x, Alpha Radar 5x,
  MarketBrew 3.3x, X API MCP 1.8x
- **7 serve for free despite listing a price** — mispricing in the other
  direction, and presumably nobody has told them

Cost: zero. Nothing was signed. 95 probes in ~7s.

**Methodology caveat, recorded in the report itself:** this probes the *first*
endpoint-bearing service per agent, not all 395. An agent whose first service is
dead may have live ones. The honest claim is "43% of endpoint-bearing agents have
a dead first endpoint", not "43% of services are dead". Do not let this get
rounded up in a thread.

### Availability is not stable within an hour

Predict Protocol #2134 answered a conformant 402 at 19:13 and was consistently
unreachable across five probes from ~19:20 — its host stopped resolving mid-session.
Three consecutive full index runs after that were stable 15/15, so this is not

I think this is the strongest evidence we have for the product's own thesis, and

could have called the vendor directly". The measured answer is that a buyer doing

## 2026-07-21 — asset and network authorization, and why I did not take the review's advice literally

Codex C1: "live asset/network/scale authorization is not closed". The allow-list
was optional, there was no network allow-list at all, and an offer that declared
no decimal scale proceeded on the caller's assumption.

The first two are straightforwardly right.

**Network allow-list.** An asset allow-list without a network allow-list is
half a check. "15 units of token X" is meaningless without the chain: a
contract at a familiar-looking address on a chain we never meant to touch is a
different asset, and the cap arithmetic cannot tell — base units are integers.
`selectOffer` now filters on both, and it filters *before* choosing the cheapest
entry rather than checking the winner afterwards, so a challenge offering a bad
cheap entry and a good expensive one is still payable on the good one.

**Both lists are now mandatory in real mode.** The procurer refuses to start
with `REAL_PAYMENTS_ENABLED=true` and either list unset. Verified by starting
it: refuses with neither, refuses with assets only, starts with both.

### Where I disagreed, and why

The review's third point was: require a declared decimal scale whenever real
money moves. I implemented that, then checked it against the marketplace scan
before trusting it.

**41 of the 47 live x402 vendors declare no `extra.decimals`** — including
OKLink #2023, the vendor G1 and G2 both actually paid. That rule would have cut
the hireable pool from 47 to 6 and broken the one payment path with real
transactions behind it. It would have looked like a security improvement and
functioned as a product outage.

The check is not really about the vendor declaring the scale. It is about *us
not knowing it*. So the rule became: the scale must be **known**, not
**declared**. An undeclared scale is safe exactly when the asset and network are
both explicitly allow-listed, because then the scale is a fact about a token we
chose in advance rather than a claim the vendor is making. USD₮0 on X Layer is
6 decimals whether or not the seller says so.

Since `selectOffer` has already refused anything outside both lists, reaching
the check with both configured means the asset is one we named ourselves. The
allow-list *is* the decimals assertion.

Four tests pin the corners: refused with no lists, refused with only the asset
list, paid with both — that last one is the OKLink case and it must keep
working — and simulation left permissive so the fixture vendors are unaffected.

Confirmed against the live vendor rather than a fixture: OKLink's challenge
selects at 15 base units, USD₮0, eip155:196, `declaredDecimals: null`, payable
under both lists; and refused outright when the network list says eip155:1.

### Operational consequence, for whoever fires the next live payment

`X402_ALLOWED_NETWORKS=eip155:196` is now required alongside
`X402_ALLOWED_ASSETS`. The G1/G2 command needs it added or the procurer will
not start. Documented in .env.deploy.example and docker-compose.deploy.yml.

Tests: 95 procurer, 36 gateway, 38 pytest.

## 2026-07-21 — the stale-reclaim race, which the fault-attribution fix had already half-closed

Codex C1, the last open finding. Worth writing up carefully because the review's
diagnosis was right and its severity was already out of date by the time I got
to it.

### The mechanism

`claim_next_task` reclaims any job whose `updated_at` is older than
`worker_stale_after_seconds`. That is how a crashed worker's job gets picked up
again, and it is correct. Its safety rests entirely on a *live* worker bumping
`updated_at` more often than the window — every `store.transition()` does, so
the real question is the longest gap between two transitions.

Codex found that gap: a non-ok vendor response updated performance and wrote no
checkpoint. Five sequential 60-second vendor timeouts therefore produced no
heartbeat at all and walked the job straight past the 300-second window while
the first worker was still running it. A second worker then claims it, restarts
at planning, and both hire.

### Why it is less severe than it reads

The procurer's idempotency on `(task_id, subtask_id, vendor_endpoint)` absorbs
most of the damage: the second worker replays the recorded receipt instead of
paying again. But only while both workers walk the same candidate order, and
they need not — sourcing re-ranks against `vendor_performance`, which the first
worker is concurrently mutating. Different order means a different vendor, a new
idempotency key, and a genuinely second payment.

So it is a real money bug, just a narrower one than "two workers, double spend".

### It is already mostly fixed, by accident

The fault-attribution change earlier today added `store.transition()` to **both**
branches of the non-ok vendor path — precisely the missing checkpoint Codex
identified. Every candidate outcome now heartbeats, so the longest gap between
transitions is one vendor call (~65s against a 300s window) rather than the
whole candidate list.

I only noticed this while reading the review's line references against the
current code. Worth recording: two findings that looked independent shared a
root cause, and fixing the honesty bug fixed most of the concurrency bug.

### What was left, and what I did

A configuration hole. The invariant "window > longest single node" was nowhere
written down and nothing enforced it, so raising `VENDOR_TIMEOUT_MS` past the
stale window silently reopens the race. That is the worst kind of bug to leave:
invisible, config-only, and it surfaces as an unexplained double payment.

`assert_stale_window_is_safe` now runs at worker startup and refuses to start
below 3x the longest call. Default raised 300 -> 900 for slack. The invariant is
documented on the setting itself.

**I got the test wrong first.** I asserted that 300s should be rejected, which
failed — 300 > 65*3. My own threshold said 300 was safe and I had written the
test from the review's framing instead of from the arithmetic. Corrected to test
what the invariant actually claims: 60s and 120s are refused, 300s passes *but
only because the checkpoint fix landed*, and the shipped default has room to
spare. Better to have the test disagree with me than to have shipped a comment
asserting something false.

### Lane

`apps/firm` again — third crossing, same justification: money safety, small,
externally found. Ishita should review. 42 pytest (was 41).

**All Codex findings are now closed** except the ones I deliberately left, which
are recorded in the review document as disclose-rather-than-fix.

## 2026-07-21 — the worker can now be caught hanging

Tier 1 of the post-Codex list. The worker is the process nobody watches: the
gateway takes the money, the procurer moves it, and neither turns a paid job
into a deliverable. If this loop stops, the visible symptom is nothing at all —
jobs sit at `paid`, the gateway keeps accepting payments, and from a buyer's
side that is indistinguishable from an agent that took the money and vanished.

`restart: unless-stopped` does not cover it. Fly restarts a process that
*exits*; a hang is not an exit.

### The trap, avoided deliberately

A health endpoint served from its own asyncio task answers 200 happily while
the loop it claims to represent is wedged, because serving HTTP and claiming
jobs are different coroutines. That check would have been worse than none: it
would have reported healthy through the exact failure it existed to catch.

So health is derived from the loop rather than the process. The loop stamps a
heartbeat each iteration; the endpoint reports how stale that stamp is and
returns 503 once it exceeds the threshold.

The threshold is `worker_stale_after_seconds` — the same constant that governs
job reclamation, on purpose. It makes two things coincide that always should:
the moment this worker is declared unhealthy is the moment another worker
becomes entitled to steal its job. Two different numbers would open a window
where a job is reclaimed from a worker still considered healthy.

### The bug the deploy found

First deploy failed: `1 total, 1 critical`, health check never passed, while the
server was running perfectly.

`asyncio.start_server(host="::")` sets `IPV6_V6ONLY`. Confirmed from inside the
container: `::1` accepted, `127.0.0.1` refused. Fly's platform check arrives on
IPv4, so it could not reach a server that was working. **A health check failing
against a healthy service is the worst variety** — it would have restart-looped
a fine worker, and the "fix" of relaxing the check would have removed the value.

Now built explicitly with `socket.create_server(..., dualstack_ipv6=True)`, so
6PN (IPv6) and platform checks (IPv4) both reach it. Pinned by a test that
connects over both families.

### Verified on the deployed machine

```
checks 1/1
{"ok":true,"service":"firm-worker","uptime_seconds":30.4,
 "seconds_since_tick":1.6,"stale_after_seconds":900,"ticks":30}
```

`ticks` is the useful field for an operator: a flat count across two checks
means the loop is alive but making no progress, which no liveness probe can
distinguish on its own.

Machine-level `[checks]` rather than `[[services]]`: this needs no published
port, and a worker reachable from the internet would be a spending-adjacent
surface for no reason.

explicit request.

### Answered same session: Poulav chose (3b)

The OKX CLI is out of the money path. Payments and refunds both sign in-process
with viem from `0xC029…50e0`, so the wallet that receives customer payments is
the wallet that pays vendors and the wallet that refunds. The container is now
self-sufficient: its only external dependency is an RPC endpoint.

**The EIP-712 domain is proven, never assumed.** This was the real risk in the
change. A wrong domain does not error — it produces a well-formed signature the
vendor accepts, relays, and fails to redeem, so the money never moves and the
failure surfaces far from its cause. USD₮0's `name()` is `USD₮0` with U+20AE,
not an ASCII T; `USDT0` is the obvious guess and is wrong.

So `chain.ts` reads the token's `name()`, collects candidate versions (operator
override, on-chain `version()`, the vendor's `extra`, then "1"/"2"), and keeps
only the candidate whose hash reproduces the token's own `DOMAIN_SEPARATOR()`.
No match means refuse to sign. Verified on chain:

```
name()              USD₮0        (U+20AE, not ASCII T)
DOMAIN_SEPARATOR()  0xd591d9baf744328d9400b923cb02c9474d367d591ca1ab24d8c4068be527599d
matched domain      name=USD₮0 version=1 chainId=196
typehash            0x7c7c6cdb…1a2267 == canonical EIP-3009
```

**The nonce is derived, not random.** `keccak256(wallet + idempotency key)`, so
re-signing a subtask reproduces the *same* authorization. EIP-3009 nonces are
single-use on-chain, which makes the token itself the enforcer of at-most-once
payment — a bug in our retry logic becomes a harmless duplicate instead of a
double-spend. Previously that safety lived entirely in our own bookkeeping.

**Redeemability is verified against the live token, not asserted.**
`pnpm -F @firm/procurer verify-signing` drives the shipped `localSigner` and
static-calls `transferWithAuthorization` on X Layer — no broadcast, no gas, no
state change. It also runs a negative control, because a positive result alone
would prove nothing:

```
valid signature   : ACCEPTED by the token
tampered signature: reverted -> TetherToken: invalid signature.
```

**Refunds changed shape.** A refund is now a transaction we broadcast rather
than an authorization someone else redeems, so the wallet needs native gas.
Checked before sending; an out-of-gas refund would otherwise surface as an
opaque RPC error at the exact moment the guarantee is invoked. Observed: 6.0e12
wei at 2.0e7 wei/gas ≈ four refunds' headroom. The receipt is awaited and
`status: success` required, because a reverted transfer still returns a hash and
would otherwise be reported as a completed refund.

`REFUND_FROM_ADDRESS` survives with a changed purpose: it no longer guards an
unverifiable external account, it guards the wrong key being deployed.

Deleted `src/signer.ts`. `src/signer.test.ts` never tested it — it tested x402
against the real OKLink challenge — so it was renamed `oklink-challenge.test.ts`
and kept. 118 procurer tests (was 100), 20 more against real Postgres, 49

### 2026-07-22 — the inbound settle path was BROKEN, not merely unproven

Found while preparing a QA purchase of Firm Express. Poulav approved the test

```

   30001 "incorrect params"
B) {x402Version, paymentPayload, paymentRequirements}
   isValid: true, payer 0x212e82dc…
```

**The gateway sent the base64 header string; the facilitator wants the decoded

  moved"* — the same wording that has already rejected this team twice.

that would be rejected anyway.

### Also found: the challenge advertised four times what it accepts

While verifying step 4, the live challenge showed `outputSchema.input` as a bare

### The 402 named no URL to replay against

Prompted by OKX (@BunsanXBT) publishing the three most common listing blockers.
Checked all three rather than assuming, and the third turned up a real defect.

Blockers 1 (compliant 402 advertising `eip155:196`) and 2 (reachable endpoint)

Reading OKX's A2MCP guide against our own output found the defect. It specifies
a top-level `resource` object that "contains `url`, `description`, and

```

OKLink #2023    top-level resource {url, mimeType}   accepts[0].resource null
Firm            top-level resource ABSENT            accepts[0].resource "firm:express:market_snapshot"
```

We published the exact inverse of both, and our value is not a URL. **Nothing in

contributor to David's "paid replay returns no content", and `x402-check` passes

reopening the settle path that broke twice. Dropping `accepts[].resource` to
match the vendors exactly remains available and should be tested against a live
settle, not assumed.

The URL is an env var, deliberately not derived from the request `Host` header —
that is caller-controlled, and this field tells a buyer where to send money.
Unset omits the object rather than emitting a guess: a wrong replay URL is worse
than an absent one, and that case is pinned by a test.

`[x402Version, resource, accepts]`, the same shape OKLink emits. 87 gateway
tests, was 83.

## 2026-07-23 — Projects delivered end to end on real money, and the CLI-shape bug it surfaced

### The bug a live purchase caught that tests did not

The first paid Projects attempt failed with `merchant returned HTTP 400`,
`INVALID_ARGS`. No money moved (`txHash: null`) — the authorize→validate→settle
guard refused before settlement, exactly as designed.

Cause: the OKX payment CLI sends business params as `--param key=value`, flat
strings only, with no nested-object form. A buyer's
`--param budget_cap={"amount":"1000000",...}` reaches the server as
`{ budget_cap: "{...}" }` — `budget_cap` a STRING — and the Zod object schema
rejected it with 400. That is the "paid replay is not a valid x402 call"
rejection class this entry has already been hit with twice, and it is the way a
reviewer buying Projects through the CLI would first meet it. `x402-check` never
caught it because it sends a well-formed JSON body, not the CLI's flat params.

Fixed server-side in `coerceProjectArgs`: a JSON-string `budget_cap` (or
`constraints`) is parsed back to an object before validation; a direct JSON POST
is untouched, and a non-JSON string is left in place so the schema still rejects
genuine garbage. Gateway 103 tests (was 97). Verified live before spending: the
CLI-shaped body now returns 402, not 400.
