## 2026-07-20

Done:
- Recorded the verified OKX buyer-flow shape (against `onchainos 4.2.6`, installed locally)
  in docs/firm/LIVE_PAYMENT_RUNBOOK.md §1, including where it contradicts INTERFACES §5.
- Implemented the real x402 buyer flow in packages/procurer:
  - `src/x402.ts` — pure challenge parsing (v2 `PAYMENT-REQUIRED` header, v1 body),
    offer selection, legacy `X-PAYMENT` assembly, `PAYMENT-RESPONSE` decoding.
  - `src/signer.ts` — the only key-touching code; shells out to
    `onchainos payment pay-local`, passing FIRM_WALLET_KEY as `EVM_PRIVATE_KEY`
    in the child env only (never argv, never a file, never a log line).
  - `src/vendor.ts` — probe -> parse -> verify caps -> sign -> replay -> capture receipt.
  - Gated behind REAL_PAYMENTS_ENABLED (default false). Simulation still runs through
    the identical reservation/cap machinery, so the money bookkeeping under test is
    the code that runs live.
- Fixed two real defects found while reading my own lane:
  1. `pg.Pool.query()` does not pin a connection, so the intended BEGIN/COMMIT could
     land on different sockets. Added a shared pool plus `withTransaction`, which
     checks out one client.
  2. Cap and idempotency checks were check-then-insert. Two concurrent calls could
     both pass a cap, or both pay. Replaced with reserve-before-pay: a
     `pg_advisory_xact_lock` serialises every spend decision, the caller's
     `max_amount` is reserved BEFORE the vendor is probed, and the row is settled to
     the vendor's actual price afterwards.
- Added a `state` machine to `procurer_calls`: reserved -> signed -> settled, plus
  released. A call that fails BEFORE signing releases its reservation and is
  retryable; a call that fails AFTER signing does not, and a retry of that key
  returns REQUIRES_HUMAN rather than signing a second authorization for money that
  may already be gone.
- Implemented the /refund policy that was entirely missing: auto-approve up to the
  task's quoted price, REQUIRES_HUMAN above it, daily refund cap, per-task
  idempotency. The quoted price is read from `firm_jobs` (read-only, cross-lane),
  never from the request body — taking it from the request would let the caller
  authorise its own refund.
- GET /caps now reports spent_today, refunded_today, and unconfirmed_signatures.

Verified:
- packages/procurer tsc --noEmit: clean.
- packages/procurer vitest: 46 tests pass, including 13 concurrency/cap tests against
  a real Postgres (PROCURER_TEST_DATABASE_URL).
- Negative control: with the advisory lock removed, 3 of those tests fail, one of them
  a genuine 300k spend against a 200k daily refund cap. The tests do prove the property.
- Full eval suite (5 files, 10 tests) passes with these changes.
