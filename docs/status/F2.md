## 2026-07-21 — the gateway never actually took the money

Codex review, C0. Verified by reading the code, then by a negative control.

`verifyPayment` asks the facilitator "is this signature valid for these
requirements". That is all it asks. It broadcasts nothing. The gate accepted a
payment on that answer alone, inserted the job, hired vendors with the Firm's
own money, and returned a `PAYMENT-RESPONSE` header saying `status: "success"`.

The buyer's authorization was never redeemed. It sat unsubmitted until it
expired. Every paid call was served free, while the response asserted a payment
that had not happened.

The test suite was green throughout, and the reason is worth writing down: the
fake facilitator answered every path with the same body, and the verify fixture
returned `transaction: "0xsettled"`. A verification response carries no
transaction — a real facilitator has nothing to put there. The fixture invented
the field that would have made the gap visible, and then the assertion on it
read as proof that payment had settled. A test that passes for the wrong reason
on the money path is worse than no test, so the fake now records the request
path and the verify fixture no longer pretends.

### Fixed

`settlePayment` posts to `/settle` and is now what gates the job. Order is
verify -> settle -> insert, and settlement runs before a single vendor is hired,
because hiring spends our money: verifying, spending, and only then finding that
settlement fails would mean absorbing the whole job for a buyer who never paid.

It refuses to report success without a transaction reference. "Settled, but we
cannot tell you the tx" is indistinguishable from "not settled", and the
`PAYMENT-RESPONSE` would be asserting a payment we cannot evidence.
`encodeSettlement` now takes a settlement rather than a verification, so that
header carries a real hash or the request fails.

Also closed, same change (Codex C1): a settled payment with no payer address is
refused. We would be holding the buyer's money with no idea where to send it
back, and the fallback was a placeholder address — i.e. a stranger. The refund
guarantee is absolute, so a job we could not refund is a job we do not take.

### Tests

Three new live tests against a real gateway process, a real Postgres and a
facilitator we control: settle-then-insert with the tx in the header and the
payer on the job row; verify-succeeds-settle-declines writes no job; settlement
without a transaction writes no job. Five new unit tests on `settlePayment`.

Negative control run: removing the settle guard fails exactly the two tests that
name the behaviour, and nothing else. They are not vacuous.

36 gateway tests (was 28).

### Still unverified, and it gates the listing

TODO(unverified): `/verify` and `/settle` route names and body shapes are
confirmed only against our own fake. The real OKX facilitator may use different
routes, require authentication, or expect a different envelope.
`X402_FACILITATOR_URL` gates the entire path and is unset, so the gateway

## 2026-07-21 — the facilitator, found and wired

`X402_FACILITATOR_URL=https://web3.okx.com/api/v6/pay/x402`

Documented in OKX's HTTP API reference, not guessed:
`POST /verify`, `POST /settle`, `GET /settle/status`, `GET /supported`.

### It found a bug before it found a success

Our code built the route with `new URL("/verify", base)`. A leading slash resets
to the root, so with a base carrying a path prefix every call would have gone to
`https://web3.okx.com/verify`.

The failure mode is the expensive kind. A 404 from that URL surfaces through our
own error handling as "the facilitator rejected the payment", so we would have
spent the debugging effort on the payload and the signature while the path was
wrong the whole time. `facilitatorUrlFor` joins without discarding the prefix.

### Authentication

Every facilitator endpoint is an authenticated OKX API: HMAC-SHA256 over
`timestamp + METHOD + requestPath + body`, base64, with `OK-ACCESS-KEY`,
`OK-ACCESS-SIGN`, `OK-ACCESS-PASSPHRASE` and `OK-ACCESS-TIMESTAMP`.

Three details each fail with the same opaque auth error, so each is pinned by a
test: `requestPath` is path + query and NOT the full URL; the method is
upper-cased; and the body signed must be the exact string sent, so it is
serialised once and both signed and posted from that one string.

**These are API credentials, not wallet material.** They authenticate us to OKX
so it will verify and settle a buyer's authorization. They cannot move the
Firm's funds and they sign nothing on-chain, which is why they can live in the
gateway at all — the wallet key stays in packages/procurer. Worth stating
plainly, because "the gateway now has OKX credentials" reads like a violation of
the quarantine rule and is not one.

### Verified against the real endpoint

```
GET /api/v6/pay/x402/supported  ->  200 {"code":0,...}
  {"network":"eip155:196","scheme":"exact","x402Version":2}   <- what we advertise
```

Authenticated on the first attempt, and the response confirms `exact` on X Layer
is supported — so the scheme in our challenge is one the facilitator will settle,
rather than one we assumed.

Then, against the deployed gateway with the real facilitator configured, a
forged payment header:

```
{"error":{"code":"PAYMENT_REQUIRED","detail":"facilitator rejected the payment"}}
```

That is OKX's facilitator refusing it, not our own fail-closed path. The two are
distinguishable in our error handling: an auth failure returns
`facilitator returned HTTP 401`, so a *rejection* means HTTP 200 and a real
evaluation. Auth worked; the forgery was caught by OKX.

`x402-check` still returns `valid: true` afterwards.

### Still not proven

No real buyer has completed a purchase. What is proven is that we authenticate,
that our scheme is supported, and that an invalid payment is refused. What is
not proven is the success path: a valid authorization verifying, settling, and
returning a transaction reference.

The remaining risk is concentrated in one place. These docs describe the batch
(`aggr_deferred`) flow, where `/settle` means *accepted* rather than *landed* and
the caller polls `/settle/status`. `settlePayment` refuses to report success
without a transaction reference, which is right for `exact` and would reject a
valid batch acceptance. We advertise `exact`, so it should not bite — but it is
untested, and it is the one place where correct-looking code could still refuse
a paying customer.
